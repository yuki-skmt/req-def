# 要件定義とアーキテクチャ設計講座
## 1日目
### システム開発工期遵守状況・品質状況
- 500人月以上の大規模なシステム開発においては、42.3%ものプロジェクトが「予定より遅延」となり、品質においても26.1%が「不満」という結果に終わっている。

- 工期の遅延理由の分析では、要件定義における問題が遅延の原因の55％を占めると言われている。

- 主な失敗要因の一つとして、合意形成にどれだけの時間がかかるかを意識できないことが挙げられる。
  - 要求は膨らむものとして臨むべき。また、追加要求には慎重になるべき。

- 顧客は見たり、触ったりしないと適切に判断できないことも念頭に置く必要がある。
  - 図示の重要性。

- 見積時期とリスク
  - システム開発の初期ほど、見積もり上の規模と最終的な規模に乖離があるため、高いリスクがあると言える。
  - 工期の終盤に近づくにつれ、情報の充実などによって適正な規模に収束する。

### As-Is(現状)からTo-Be(将来)への流れ
- 現状の業務がどうなっており、そこからどのように計画・開発・移行するか、また最終的にどのような形になるのか、といったことについて業務の視点とITの視点に分けて絵で見せることが重要。

- IT経営プロセスと要件定義
  経営戦略の策定には、システムのステークホルダ（システム開発によって影響を受ける人)を意識する必要がある。
  - エンドユーザから株主まで対象は様々。（想定のしやすさも様々）

### 罠になりやすい非機能要件
- 非機能要件は顧客も認識できていないことが多いうえ、問題が顕在化した時に大きな問題になりやすい(セキュリティなど)。

- 結果的に、問題発生時に満足度が下がる要因となる。

- 顧客からすると可用性を高めるための二重化などは平時はコストでしかなく、開発側にとってもコストがかかったり、難易度が高かったりするので敬遠されがちである。
  - どちらからも避けられがちだからこそ、適切に非機能要件を見極める必要性がある。
### 要件定義の手法
- 図の表記方法
  - 顧客が理解・推測できるか、将来のシステムが見えるかが重要。
  - 要件定義段階で図を作成しておけば、粒度を落としてシステム設計に活かすという使い方もできる。

- ヒアリング技法
  - 隠れた要求事項を引き出す。(「隠れた」の部分が本質)
  - 「必要な機能を言ってください」では、顧客は思いつくものしか言えない。
  - 顧客がまとめてくるものは穴だらけと思ってかかること。
  - 顧客企業で常識としてまかり通っているものは絶対に要件として出てこない。
　 - 曖昧性を排除するプロセスに「業務知識(業務ルール・専門用語・トレンド)」が必要となってくる。

- 上記の技法(要件定義のスキル)は全て「コミュニケーション能力」を基盤とするものである。
  - 傾聴力を鍛えておくこと。
  - ビジネス要求は雑談から得られることもある。

- 上流工程(要求分析・要求工学)の知識体系にも目を通しておくといい。
  - REBOKなど
### 企業とは
- 「ゴーイングコンサーン(継続企業の前提)」、「利益追求・拡大」が企業の必須条件である。

### 目的と手段の関連づけ
- 大きい目的(経営レベル)があり、それを分割した小さい目的(現場レベル)があり、小さい目的を解決するための手段がある。
  - 要件定義で明確にすべきは、「真の目的」つまり、経営レベルでの目的。

### 企業が変わるべき理由
- 外部環境の要因(コロナ、円高・円安etc)や、内部資源の要因(資金、人材etc)。
- 「現状(As-Is)」と「あるべき姿(To-Be)」との差をどう埋めていくかが経営課題となる。
  - ただし、現場レベルの要求と経営レベルの要求が食い違うことがままあるので注意。

### CSFとKGI、KPI
- 計測できないものに関しては評価ができない。
- 評価のための定量的な尺度を与えるのがCSF・KGI・KPIの意義。
  - 従ってこれらの尺度に関しては、可能な限り具体的であることが求められる。
  - 要件定義の場では、抽象性・曖昧性をいかに排除するかがカギとなる。

### ステークホルダの定義
- 「システムに関連する個人・組織すべて」をステークホルダとする。
  - ステークホルダの中でも、「誰が決定権者か」というのはとりわけ重要。
  - 決定権者以外と話しても進まないことも多い。

### 全社戦略
- PPM（プロダクトポートフォリオマネジメント）…花形・問題児・金のなる木・負け犬のアレ。
  - ファイブフォース分析…外的環境の分析に使用する。
  - SWOT分析…強み・弱み・機会・脅威のアレ。
### 情報システム戦略の目的
- 経営戦略に沿って情報システム戦略を策定する。

### テクノロジアーキテクチャ
- 業務を実現するためのハードウェア、ソフトウェア、ネットワークなどの技術を体系化したもの。
  - クラウドの隆盛により変革の可能性がある。
